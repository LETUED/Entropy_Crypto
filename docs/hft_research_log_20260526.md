## 1. HFT 프로젝트 배경 및 목표

### 1.1 백테스트 전략과의 관계

백테스트 연구(Exp01~Exp14)는 168H(주간) 타임스케일의 MPE 신호를 검증한다.  
그 결과 핵심 발견:

- **저엔트로피 구간에서 롱 진입** → 승률 69.9%, 평균 PnL +0.649% (Exp08)
- 단, 4년간 거래 횟수 ~20건 → 통계적 유의성 부족

HFT 프로젝트는 **동일한 엔트로피 철학을 tick 레벨로 이식**해 거래 빈도 문제를 해결하려는 시도다.

### 1.2 HFT 핵심 가설

```
백테스트(주간): 엔트로피 낮음 = 시장 질서 있음 = 방향성 편향 존재
HFT(tick):    엔트로피 낮음 = 매수/매도 쏠림 = 단기 방향성 존재
```

두 스케일에서 같은 원리가 작동한다는 가설.

### 1.3 HFT 프로젝트 목표

| 목표 | 내용 |
|------|------|
| **고빈도** | 하루 수십~수백 거래 기회 |
| **24/7 운용** | 시간대 무관, 자동화 가능 |
| **범용성** | 코인/주식 모두 적용 가능 |
| **통계적 우위** | 카지노 하우스 엣지 구조 (소량 × 빈도) |

### 1.4 참고 논문

| 논문 | 핵심 내용 |
|------|---------|
| [arXiv 2312.16637](https://arxiv.org/abs/2312.16637) | 초고빈도에서 가격 예측 가능성 실재 |
| [arXiv 2512.15720](https://arxiv.org/pdf/2512.15720) | **엔트로피는 이동 방향이 아닌 크기를 예측** |
| [arXiv 2507.22712](https://arxiv.org/html/2507.22712v1) | 오더북 필트레이션 + 방향 신호 추출 |
| [PMC 10813935](https://www.ncbi.nlm.nih.gov/pmc/articles/PMC10813935/) | 오더북의 열역학적 분석 |

> **핵심 논문 인사이트 (2512.15720):** 엔트로피 신호는 "이동 방향"이 아닌 "이동 크기"를 예측한다. 따라서 Maker 전략(방향 무관, 스프레드 수집)이 이론적으로 더 적합하나, 방향 신호(OBI)와 결합 시 Taker로도 활용 가능.

## 2. HFT 아키텍처 설계

### 2.1 신호 체계

```
신호 1: Shannon Entropy (타이밍)
  소스: Binance aggTrade 스트림 (매수/매도 시퀀스)
  계산: p_buy = 매수 비율, H = -p*log2(p) - (1-p)*log2(1-p)
  신호: H < 0.85 → 단방향 쏠림 (70%+ 한쪽 방향)
  의미: 정보 거래자 활동 가능성 ↑, 예측 가능 구간

신호 2: Order Book Imbalance (방향)
  소스: Binance depth20@100ms 스트림 (상위 10레벨)
  계산: OBI = (bid_vol - ask_vol) / (bid_vol + ask_vol)
  신호: OBI > +0.15 → 매수 압력 / OBI < -0.15 → 매도 압력
  의미: 현재 시장 방향성
```

### 2.2 진입 조건 (Taker 기준)

```python
진입 조건 (전부 동시 충족):
  1. H < 0.85          # 엔트로피 신호 (예측 가능 구간)
  2. |OBI| > 0.15      # 방향성 있음
  3. |OBI| <= 0.50     # 극단 아님 (Adverse Selection 차단)
  4. 신호 첫 발생 시   # 기존 신호 지속 중 중복 진입 금지
  5. (HFT-Exp03) trades/sec >= 1.5  # 거래량 게이트
```

### 2.3 청산 조건

```
우선순위:
  1. 목표가 도달 (+0.2%)          → TARGET 청산
  2. 스톱가 도달 (-0.1%)          → STOP 청산
  3. 엔트로피 정상화 (H >= 0.85)  → ENTROPY_EXIT 청산
  4. 60초 타임아웃                → TIMEOUT 청산
```

### 2.4 공통 파라미터

```python
# hft/config.py 기준
SYMBOL              = "SOLUSDT"
TOTAL_CAPITAL_USDT  = 10_000.0
TAKER_POSITION_PCT  = 0.15      # 포지션당 15% = 1,500 USDT
TAKER_MAX_POSITIONS = 3         # 최대 동시 3개
TAKER_TARGET_PCT    = 0.002     # +0.2% 목표
TAKER_STOP_PCT      = 0.001     # -0.1% 스톱
TAKER_MAX_HOLD_S    = 60        # 최대 보유 60초
ENTROPY_WINDOW      = 100       # 마지막 N 거래
ENTROPY_THRESHOLD   = 0.85      # Shannon 엔트로피 임계값
OBI_BOOK_LEVELS     = 10        # 오더북 상위 N 레벨
OBI_THRESHOLD       = 0.15      # 방향성 최소 임계
OBI_ACTIVATE_MAX    = 0.50      # 극단 Adverse Selection 차단
MAKER_FEE           = 0.0002    # 0.02%
TAKER_FEE           = 0.0005    # 0.05%
```

## 3. HFT-Exp01: Maker 시장조성 — 5세그먼트

**파일:** `run_hft_paper.py`, `hft/simulation/paper_trader.py`  
**날짜:** 2026-05-26  
**심볼:** BTCUSDT → SOLUSDT (테스트)

### 설계

```
총 자본 10,000 USDT를 5세그먼트(각 2,000 USDT)로 분할
각 세그먼트가 독립적으로 Maker 호가를 제출
  - 엔트로피 신호 발생 시 세그먼트 활성화
  - OBI 방향에 따라 매수/매도 호가 기울임
  - 세그먼트당 2% 스톱로스

스프레드 설정: ±0.003% (중간가 기준 총 0.006%)
```

### 실험 경과

**초기 문제 1 — 스프레드 너무 넓음:**
- 초기 MAKER_SPREAD_PCT = 0.0004 (±0.04%)
- BTC 기준 호가가 중간가에서 $31 이탈 → 체결 불가
- 수정: 0.00003 (±0.003%)으로 축소

**초기 문제 2 — Adverse Selection:**
- OBI = -0.999 (강한 매도 압력)에서도 세그먼트 활성화
- 정보 거래자에게 역방향 유동성 제공 → 즉시 재고 손실
- 수정: `OBI_ACTIVATE_MAX = 0.5` 추가 (|OBI| > 0.5이면 활성화 차단)

### 결과

| 항목 | 값 |
|------|---|
| 테스트 시간 | UTC 02:00~02:10 (10분) |
| 거래량 | 약 30 거래/분 (새벽 quiet) |
| 세그먼트 활성화 | 산발적 |
| 실현 PnL | 음수 |

### 핵심 발견

**구조적 수익 불가 확인:**

```
스프레드 수익: ±0.003% × 2 = 0.006% (왕복)
수수료:       0.02% × 2 = 0.04% (왕복)
손익:         0.006% - 0.04% = -0.034% (거래당 손실)
```

SOL/BTC의 실제 bid-ask 스프레드는 0.01~0.02%. 이 스프레드 안에 들어가서 체결을 받으려면 스프레드 폭을 좁혀야 하지만, 좁힐수록 수수료 대비 수익이 마이너스가 된다.

**Adverse Selection 문제:**  
저엔트로피(단방향 쏠림) 구간은 정보 거래자가 활성화된 구간이다. 이때 Maker 호가를 내면 정보 거래자가 내 호가를 역방향으로 채워버린다. 엔트로피 신호는 Maker에게 "진입 신호"가 아니라 "철수 신호"여야 한다.

**결론: Maker 시장조성 전략 포기.** 소매 수준에서 수익 구조 성립 불가.  
→ 전략 전환: Taker 방향성

**파일:** `run_hft_taker.py`, `hft/simulation/paper_trader_taker.py`

**심볼:** SOLUSDT  
**테스트 시간:** UTC 02:42~02:58 (16분)

### 설계

```
엔트로피 신호(타이밍) + OBI(방향) → 시장가 주문으로 방향성 매매
  - 저엔트로피 + OBI 매수 압력 → LONG @ ask
  - 저엔트로피 + OBI 매도 압력 → SHORT @ bid
  - 목표: +0.2%, 스톱: -0.1%, 최대 보유: 60초
```

### 페이퍼 트레이딩 결과

|------|---|
| 테스트 시간 | UTC 02:42~02:58 (16분) |
| 거래량 | 0.88 거래/초 (새벽 quiet) |
| 총 거래 | 3건 |
| 승 / 패 | 0 / 3 (0%) |
| 실현 PnL | -5.02 USDT |

### 거래별 상세

| 포지션 | 방향 | 진입 | 청산 방식 | 보유 | PnL |
|--------|------|------|---------|------|-----|
| POS0 | SHORT | 86.1300 | ENTROPY_EXIT | 6.1s | -1.67 USDT |
| POS1 | LONG | 86.1400 | ENTROPY_EXIT | 4.7s | -1.85 USDT |
| POS2 | SHORT | 86.1200 | ENTROPY_EXIT | 36s | -1.50 USDT |

### 핵심 발견

**POS1 분석 (방향은 맞았음):**
```
POS1: LONG @ 86.14  
진입 후 가격: 86.15 → 86.16 → 86.17 (상승 ✅ 방향 정확)
uPnL: +0.09 USDT (진행 중)
ENTROPY_EXIT 발동: 4.7초 만에 종료 → 최종 PnL: -1.85 USDT
```

방향은 맞지만 4.7초 안에 엔트로피가 정상화되어 이탈.

**ENTROPY_EXIT 분석:**
- 모든 손실이 목표/스톱이 아닌 ENTROPY_EXIT에서 발생
- 원인: 0.88 거래/초에서 100-window = 113초 소요
- 단발성 burst(4~7초)가 entropy 신호를 발생시키지만 price가 0.2% 도달 전에 종료

**시간대 진단:**
```
UTC 02:42 = KST 11:42 AM (한국 오전)
SOL 거래량: 0.88/s (피크 대비 1/5 수준)
→ 신호 지속 시간이 너무 짧음 (피크 시 5배 빠른 갱신)
```

## 5. HFT-Exp03: Volume-Adaptive Window

**파일:** `run_hft_taker.py` + `hft/signals/entropy_rt.py` (개선)  
**날짜:** 2026-05-26  
**심볼:** SOLUSDT  
**테스트 시간:** KST 14:33~14:59 (26분)

### 설계 변경 — Volume-Adaptive Window

**문제:** 거래량에 따라 동일한 window(100)의 시간적 의미가 달라짐
```
새벽 0.88/s  → window 100 = 113초 → 엔트로피가 느리게 변함
피크 5/s     → window 100 = 20초  → 엔트로피가 빠르게 변함
```

**해결:** 항상 30초 분량의 거래를 window로 사용

```python
# hft/config.py 추가 파라미터
ADAPTIVE_WINDOW_SECONDS = 30   # 목표 time span
ADAPTIVE_WINDOW_MIN     = 30   # 최소 window
ADAPTIVE_WINDOW_MAX     = 300  # 최대 window
VOLUME_GATE_MIN_TPS     = 1.5  # trades/sec 미만 → 진입 차단

# 계산 방식 (entropy_rt.py)
tps = 최근 10초 거래 횟수 / 경과 시간
window = clip(int(tps × 30), min=30, max=300)
```

**표시 추가:**
```
[14:35:08] Price: 84.7900 | Vol: 4.7/s W:141      | PE: 0.929 ...
[14:34:54] Price: 84.7300 | Vol: 0.4/s W: 30 GATE | PE: 1.000 ...
                                          ^^^^
                                          GATE = 거래량 부족, 진입 차단

```

Window 일관성 확보는 성공. 그러나 가격 이동 < 수수료 구조는 동일.

## 7. 핵심 발견 및 결론

### 7.1 증명된 사실

| 발견 | 근거 |
|------|------|
| **Shannon entropy 방향 신호 유효** | POS1(Exp02), POS4/5/6(Exp03) — 방향 맞은 경우 uPnL 양수 확인 |
| **tick 기준 가격 이동은 0.02~0.05%** | Exp03 분석: 최대 uPnL이 목표의 1/6 |
| **Maker 시장조성은 소매 불가** | spread(0.006%) < fee(0.04%) 구조적 손실 |
| **ENTROPY_EXIT 매커니즘은 작동** | 신호 지속 동안 포지션 유지, 종료 시 청산 정확 |
| **Volume Gate 유효** | 저거래량 구간 진입 차단 → 의미없는 신호 필터링 |

### 7.2 구조적 문제 분석

```
수수료 구조 (Taker 기준):
  왕복 수수료: 0.05% × 2 = 0.10%

목표 0.2% 달성 요건:
  - 60초 내 단방향 0.2% 이동 필요
  - 실측 단방향 이동: 0.02~0.05% (목표의 1/10~1/4)
  - SOL 일평균 변동성: 분당 0.01~0.05%

Maker로 전환 시 수수료:
  왕복: 0.02% × 2 = 0.04%
  → 여전히 실측 이동(0.03%) < 수수료(0.04%)
  → 수수료 절감으로 완전 해결 불가
```

### 7.3 전략 진화 경로

```
HFT-Exp01: Maker 시장조성 (이론적 최적)
  ↓ 실패: spread < fee, adverse selection
  ↓ 전환 근거: 엔트로피 = 방향 예측 가능 → Taker가 더 자연스러운 구조

HFT-Exp02: Taker 방향성 (목표 0.2%)
  ↓ 실패: 가격 이동 0.02~0.05% vs 목표 0.2%
  ↓ 신호는 맞음, 목표 및 타임스케일이 틀림
  ↓ 개선: 거래량 adaptive window 필요

HFT-Exp03: Volume-Adaptive Window 추가
  ↓ 부분 성공: window 일관성 확보, vol gate 작동
  ↓ 근본 미해결: tick 기준 가격 이동 < 수수료
  ↓ 결론: 타임스케일 조정 필요

현재 검토 중: 1분봉 기준 Maker 진입 전략
```

### 7.4 엔트로피 신호 평가

```
Shannon entropy on aggTrade stream:
  ✅ 단방향 쏠림 감지 정확
  ✅ 방향 (OBI)과 결합 시 방향 예측 55~60% (추정)
  ✅ ENTROPY_EXIT — 신호 종료 시점 정확히 감지
  ❌ 신호 지속 시간 내 가격 이동이 너무 작음
  ❌ tick 레벨에서 수수료 허들 초과 불가

→ 엔트로피 신호 자체는 유효. 타임스케일 문제.
```
